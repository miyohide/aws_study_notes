# AWS Glue

## AWS GlueのApache Spark ETLジョブにて「デバイスに空き容量がありません」

- Spark UIとAWS Glueメトリクスを使用して、Sparkエグゼキューたのデータスキューをモニタリングする
- `--write-shuffle-files-to-s3`ジョブパラメータを有効にし、ソルティング手法を使用する

## AWS GlueによるIcebergテーブルのパフォーマンスアップ

- AWS Glueデータカタログを使用して、スケジュールに基づいてIcebergテーブルの列レベルの統計を生成する
- AWS Glueデータカタログを使用して、Icebergテーブルを自動的にコンパクションする

## DynamicFrameのグループ化オプション

多数の小規模ファイルをより大きい単位で読み込み、Sparkのタスク数と実行計画のオーバーヘッドを削減する

## Icebergフレームワークを使用するApache Sparkジョブの実行

- `--datalake-formats`ジョブパラメータの値として`Iceberg`を設定
- `--enable-auto-scaling`パラメータを`true`にする

## GlueジョブのFLEX実行クラス

AWSの余剰キャパシティを利用して普段よりも安価にGlueジョブを実行できる。

## Glue DynamicFrame vs Spark DataFrame

| 項目 | Glue DynamicFrame | Spark DataFrame |
| :--- | :--- | :--- |
| **スキーマ管理** | **柔軟（自己記述型）**。実行時に推測し、型の混在や構造の変化を許容。 | **厳格（固定）**。ロード時にスキーマが確定している必要があり、不整合に弱い。 |
| **半構造化データ** | JSONなどのネストや動的なスキーマの扱いに非常に強い。 | ネストが深いデータや構造が変化するデータの扱いはやや複雑。 |
| **主な独自機能** | `ApplyMapping`（列変更）、`ResolveChoice`（型不整合の解決）など。 | `SQL`クエリ、Window関数、機械学習(MLlib)連携など。 |
| **最適化** | Glue独自のETL最適化（ブックマーク等）に特化。 | Sparkエンジン（Catalyst）による高度なクエリ最適化。 |
| **主な用途** | ETLの初期段階（データ読み込み・クレンジング）。 | 複雑なビジネスロジックの実装・高度な集計処理。 |

# Amazon Kinesis Data Streams vs Amazon Data Firehose

| 比較項目 | Amazon Kinesis Data Streams (KDS) | Amazon Data Firehose (ADF) |
| :--- | :--- | :--- |
| **主な用途** | **リアルタイム処理・カスタム分析**<br>ミリ秒単位の応答が必要なアプリ基盤 | **データロード・アーカイブ**<br>ストレージや分析基盤への自動転送 |
| **データ変換** | **コンシューマー側で実装**<br>読み取るアプリケーション側でロジックを記述 | **Lambda統合による自動変換**<br>配信過程で形式変換や加工が可能 |
| **配信・運用管理** | **シャードによる管理**<br>オンデマンドモードまたはプロビジョニングが必要 | **フルマネージド（サーバーレス）**<br>スケーリングやリトライもすべて自動 |
| **レイテンシ** | リアルタイム（通常200ms未満） | ニアリアルタイム（最小60秒または1MB） |
| **データの再利用** | **可能**<br>複数のコンシューマーが同じデータを読み取れる | **不可**<br>一度配信先へ送るとFirehose内には残らない |
| **主な宛先** | Lambda, KCL(EC2/ECS), Managed Service for Apache Flink | S3, Redshift, OpenSearch, Splunk, HTTP Endpointなど |

# Redshift

## STL_ALERT_EVENT_LOG/STL_QUERY_METRICS/STL_PLAN_INFO/STL_USAGE_CONTROL

- STL_ALERT_EVENT_LOG
    - オプティマイザが検出した問題点を記録
- STL_QUERY_METRICS
    - クエリごとのCPU使用率や実行時間などを記録
- STL_PLAN_INFO
    - クエリの実行計画の詳細を記録
- STL_USAGE_CONTROL
    - 並列実行制御の状況を記録

## VACCUME FULL/VACUUME REINDEX/VACUUME DELETE ONLY/VACUUME SORT ONLY

| オプション | 領域の解放 (Delete) | ソートの再構成 (Sort) | 主な用途・特徴 |
| :--- | :---: | :---: | :--- |
| **FULL** | ◯ | ◯ | **デフォルト設定**。削除領域の回収と再ソートを両方実施。 |
| **DELETE ONLY** | ◯ | × | 削除行の回収のみ。ディスク容量を急ぎで確保したい場合に適。 |
| **SORT ONLY** | × | ◯ | 再ソートのみ。大量データ投入後のクエリ性能改善に有効。 |
| **REINDEX** | ◯ | ◯ | **インターリーブソートキー用**。インデックスを再構築。 |

## 分散スタイル

| スタイル | データの配置方法 | 主なメリット | 適したユースケース |
| :--- | :--- | :--- | :--- |
| **AUTO** | データサイズに応じてRedshiftが自動選択。 | 運用の簡素化。 | **基本の推奨設定**。初期設計時など。 |
| **EVEN** | ラウンドロビン方式で全ノードに均等に配布。 | データが偏らない（スキューの防止）。 | 結合キーがない、または特定のキーで偏りが出る場合。 |
| **KEY** | 特定の列（分散キー）の値に基づいて配置。 | 特定の結合（JOIN）が劇的に高速化。 | 大規模なテーブル同士を共通のキーで頻繁に結合する場合。 |
| **ALL** | 全ノードにテーブルの完全なコピーを配置。 | JOIN時のネットワーク転送をゼロにする。 | 更新頻度が低く、サイズが小さいマスタ（次元）テーブル。 |

## ワークグループ

## コスト最適化

クエリ結果の再利用を使う。以前に実行されたクエリと同一のSQLが再度実行された際に前回のクエリ結果をそのまま返すことでScanオペレーションを回避し、課金を大幅に抑制させる。

## MSCK REPAIR TABLE

# 主要ストリーミングサービスの比較表

| 特徴 | **Kinesis Data Streams (KDS)** | **Amazon Data Firehose (ADF)** | **Managed Service for Apache Flink** | **Amazon MSK (Kafka)** |
| :--- | :--- | :--- | :--- | :--- |
| **主な用途** | 低遅延なリアルタイム処理 | 指定先へのデータ配信 (ETL) | 複雑なストリーム分析 | オープンソース互換の基盤 |
| **管理レベル** | プロビジョンド or オンデマンド | **完全マネージド (Serverless)** | 完全マネージド | マネージド (一部ノード管理) |
| **遅延 (Latency)** | **ミリ秒単位** | 60秒〜 (バッファリング依存) | ミリ秒単位 | ミリ秒単位 |
| **データ保持** | 最大365日間 | なし（配信のみ） | なし（処理のみ） | 長期保持可能 |
| **スケーリング** | シャード単位（手動/自動） | **フルオートスケール** | 処理能力に応じた自動制御 | ブローカー追加 |
| **実装負荷** | 高め (KCL/SDKの利用) | **低い (GUIで設定完結)** | 高め (Java/Python/SQL) | 高め (Kafkaエコシステム) |

# Lambda

## 予約済み vs プロビジョニング済み

| 項目 | 予約済み同時実行数 (Reserved) | プロビジョニング済み同時実行数 (Provisioned) |
| :--- | :--- | :--- |
| **主な目的** | **リソースの確保** と **実行制限（上限設定）** | **コールドスタートの抑制** (低レイテンシ化) |
| **動作** | 指定した数をその関数専用にキープする | 指定した数の実行環境を事前に初期化して待機させる |
| **スケーリング** | 設定値がそのまま実行数の**上限**になる | 設定値を超えてもオンデマンドでスケール可能* |
| **コスト** | **無料** | **有料** (設定した数・時間分課金) |
| **コールドスタート** | 発生する (環境は都度作成される) | **発生しない** (既に起動済みのため) |
| **設定対象** | 関数全体 | 関数の特定の「バージョン」または「エイリアス」 |

# DynamoDB

## グローバルセカンダリインデックス

デーブルのメインとなるパーティションキーおよびソートキー以外の属性を使って、データを効率的に検索するための仕組み

## ローカルセカンダリインデックス

テーブルのパーティションキーはそのままにソートキーだけを別の属性に変えて検索できるようにする仕組み。テーブル作成時のみ作成できる

# S3

## ガバナンスモード vs コンプライアンスモード

| 項目 | ガバナンスモード (Governance Mode) | コンプライアンスモード (Compliance Mode) |
| :--- | :--- | :--- |
| **主な目的** | 社内プロセスや誤操作による削除防止 | 法的規制や厳格なコンプライアンス対応 |
| **管理者による解除** | **可能**（特定の権限があれば削除可） | **不可能**（Rootユーザーでも削除不可） |
| **保持期間の短縮** | 可能（権限が必要） | **不可** |
| **保持期間の延長** | 可能 | 可能 |
| **モードの変更** | 可能 | **不可** |
